One agentic CLI, five tools at its back.

A single "delegation skill" that lets the AI CLI you have open (Claude Code or Codex) shell out to agy, kiro-cli, codex, NotebookLM, or Claude Code when the other tool is genuinely better at the task — without you leaving the terminal.

한 개의 CLI 안에서, 다섯 도구를 부린다.

지금 켜놓은 AI CLI(Claude Code 또는 Codex)가 못 하는 일을 다른 CLI (agy, kiro-cli, codex, NotebookLM, Claude Code) 에게 자동으로 떠넘기게 해 주는 단일 "위임 스킬" 입니다. 터미널을 갈아탈 필요가 없습니다.

Read this first — glossary먼저 — 용어 미리보기

A few terms used throughout this page. 이 문서에서 반복해 쓰는 단어를 먼저 정리합니다.

Agentic CLI

An AI tool that runs in your terminal. When you say "do X," it reads files, edits code, and runs commands on its own. Examples: Claude Code, OpenAI Codex CLI, Google Antigravity (agy), AWS Kiro CLI (kiro-cli).

터미널에서 도는 AI 도구. 사람이 "이거 해줘" 하면 스스로 파일을 읽고/고치고/명령을 실행함. 예: Claude Code, OpenAI Codex CLI, Google Antigravity(agy), AWS Kiro CLI(kiro-cli).

Host

The CLI you have open and are talking to right now. Usually Claude Code or Codex.

지금 당신이 켜놓고 대화 중인 CLI. 보통 Claude Code 또는 Codex.

Skill

A folder with a SKILL.md plus optional scripts. Claude Code / Codex auto-discover it and load the right one based on your request.

"특정 상황에서 어떻게 행동할지" 를 적어둔 폴더 (SKILL.md 한 장 + 부속 스크립트). 호스트가 자동으로 발견·로드.

Delegation

Instead of doing the work itself, the host shells out to another CLI and returns the result. This is what call-agent enables.

호스트가 직접 처리하지 않고 다른 CLI를 셸에서 실행시켜 결과를 받아오는 행동. 이 저장소의 핵심 동작.

MCP

Model Context Protocol — a shared protocol that lets different AI tools talk to the same external servers (files, DBs, web). Used by the kiro target's cross-registry feature.

Model Context Protocol. 여러 AI 도구가 공통으로 외부 서버(파일·DB·웹)에 접근하기 위한 규약. kiro 타깃의 cross-registry에서 등장.

RAG

Retrieval-Augmented Generation — the model retrieves source documents first and uses them as grounded evidence. NotebookLM is built around this.

Retrieval-Augmented Generation. 답하기 전에 외부 문서를 먼저 검색해 근거로 사용. NotebookLM이 잘하는 일.

Why does this exist?왜 만들었나요?

Several agentic CLIs landed around the same time. They look similar, but each is genuinely better at different things. 비슷한 시기에 여러 회사가 "내 AI CLI" 를 내놓았습니다. 다 비슷해 보이지만 각자 잘하는 게 다릅니다.

  • Claude Code — Strong at editing code, planning, and analyzing large codebases (1M-token context). Cannot generate images.
  • OpenAI Codex CLI — Has image generation (gpt-image-2) and a polished codex review for PR-level reviews.
  • Google Antigravity (agy) — Bakes Google Search grounding into the model's reasoning loop; direct access to scientific databases (gnomAD, UniProt, PubMed).
  • AWS Kiro CLI (kiro-cli) — Headless agentic terminal, natural-language → shell translation, AWS Bedrock-based second opinion.
  • NotebookLM — Drop in PDFs / URLs / YouTube and get faithful RAG answers; can produce audio overviews.
"I'm in the middle of working in Claude Code and I just need one diagram image. Now I have to open another terminal, launch Codex, copy context over… annoying."

call-agent removes that friction. Inside Claude Code you say "generate an image of …" and the call-agent skill routes to the codex target, which shells out to Codex, generates the image, and reports the path back. You never leave the host CLI.

  • Claude Code — 코드 편집·기획·1M 토큰 컨텍스트로 큰 코드베이스 분석에 강함. 이미지 생성은 못 함.
  • OpenAI Codex CLI — 이미지 생성(gpt-image-2) 가능, codex review로 PR 단위 리뷰가 강함.
  • Google Antigravity (agy) — Google Search를 모델 추론 단계에 끼워주는 grounding, gnomAD/UniProt 같은 과학 DB 직결.
  • AWS Kiro CLI (kiro-cli) — 헤드리스 자동화에 강하고, "자연어 → 셸 명령" 번역, AWS Bedrock 모델로 2차 의견.
  • NotebookLM — PDF·웹·유튜브를 통째로 넣고 정확한 RAG 답변, 오디오 개요 생성.
"지금 Claude Code로 작업 중인데 다이어그램 이미지가 하나 필요해. 일일이 다른 터미널 열어서 Codex 켜고… 귀찮네."

call-agent은 그 귀찮음을 없앱니다. Claude Code 안에서 "이미지 하나 만들어줘" 라고 하면, call-agent 스킬이 codex 타깃으로 라우팅해 Codex를 실행하고 이미지를 받아옵니다. 사용자는 CLI를 갈아탈 필요가 없습니다.

When does a skill auto-fire?언제 자동으로 호출되나요?

The call-agent router fires only under one of two conditions. No routine hijacking. call-agent 라우터는 두 가지 조건 중 하나일 때만 발동. 평상시 작업은 가로채지 않음.

① Explicit name① 도구 이름 명시

The user names the target tool — "use agy", "ask codex", "with NotebookLM". Intent is unambiguous, so dispatch.

사용자가 도구를 직접 호명 — "agy로", "ask codex", "NotebookLM에 넣어줘". 의도가 명확하니 위임.

② Capability gap② 호스트가 native로 못 함

The user asks for something the host CLI cannot do natively — image generation for Claude Code, 1M-context planning for Codex, document RAG, Google Search grounding.

호스트가 native로 못 하는 기능 요청 — Claude Code에서의 이미지 생성, Codex에서의 1M 컨텍스트 기획, 문서 RAG, Google 검색 grounding.

What it doesn't do하지 않는 것

Does not hijack "review this code" or "plan this feature". The host is presumed capable; delegation is the exception, not the default.

"이 코드 리뷰해줘", "이 기능 계획 짜줘" 같은 루틴은 그대로 호스트가 처리. 위임은 예외 상황.

Why so strict? Calling an external CLI usually spends a separate token / credit balance, slows the response down, and makes it harder for you to trace what your tool is actually doing. So the trigger keywords are deliberately narrow — delegation fires only when it clearly pays off.

왜 이렇게 보수적인가? 외부 CLI 호출은 별도 토큰·크레딧을 소비하고, 응답 속도를 늦추며, "이 도구가 지금 뭘 하고 있는지" 추적을 어렵게 만듭니다. 그래서 트리거 키워드를 좁게 잡았습니다 — 명확히 이득일 때만 발동.

Install설치

install.sh symlinks skills/call-agent into both host skill dirs (~/.claude/skills and ~/.codex/skills). Because it's a symlink, a git pull here is immediately visible to the host. install.sh는 skills/call-agent를 양쪽 호스트 스킬 폴더(~/.claude/skills, ~/.codex/skills)에 symlink 합니다. 복사가 아니라 링크라 git pull 하면 호스트도 즉시 새 버전을 봅니다.

install
$ ./install.sh                 # link call-agent into both hosts
$ ./install.sh --dry-run       # preview what would be linked
$ ./install.sh --uninstall     # remove call-agent (+ legacy *-call) links

# One skill, linked into both host dirs:
#   ~/.claude/skills/call-agent
#   ~/.codex/skills/call-agent
# Rule zero: it never delegates to the host it runs in.

Tests테스트

Four levels. Opt in to higher levels with env flags so you don't burn API credit by default. 4단계. 기본은 무료 단계만; 상위 단계는 환경변수로 켜야 동작 (실제 모델 호출 → 비용 발생).

tests
$ ./tests/run-all.sh                 # L0 + L1: binary present, --help works
$ RUN_L2=1 ./tests/run-all.sh        # adds L2: real round-trip prompts (uses credit)
$ RUN_L3=1 ./tests/run-all.sh        # adds L3: feature exercises (image gen etc.)

# Each target also has its own smoke test:
$ ./skills/call-agent/reference/agy/tests/smoke.sh

FAQ

Common questions before you install. 설치 전 자주 묻는 질문.

I don't use a host CLI — can't I just use the external CLI directly?호스트 CLI를 안 쓰는데 그냥 외부 CLI만 쓰면 안 되나요?

Yes. This repo is only useful when you want to stay inside the host CLI's flow while occasionally calling out. For standalone use, just run the target CLI directly.

됩니다. 이 저장소는 호스트 안에서 흐름을 끊지 않고 외부 CLI를 부르는 경우에만 의미가 있습니다. 단독 사용이라면 그냥 해당 CLI를 직접 쓰는 게 빠릅니다.

I don't want skills auto-firing.스킬이 자동 호출되는 게 싫어요.

Disable the skill in the host CLI's settings, or run ./install.sh --uninstall to drop the symlinks. The repo itself can stay in place.

호스트 CLI 설정에서 스킬을 비활성화하거나, ./install.sh --uninstall로 symlink만 제거하면 됩니다. 저장소 자체는 그대로 둬도 무방합니다.

Can I add a brand-new CLI?새 CLI(예: 차세대 도구)도 추가할 수 있나요?

Yes — two steps, no installer change: create skills/call-agent/reference/<new-name>/call.md, then add one row to the Route table in skills/call-agent/SKILL.md.

가능합니다 — 설치 스크립트 수정 없이 두 단계: skills/call-agent/reference/<new-name>/call.md를 만들고, skills/call-agent/SKILL.md의 Route 표에 한 줄 추가.

What about security?보안은 어떤가요?

This repo stores no tokens. Each CLI authenticates the standard way for that tool (config file or env var).

이 저장소는 토큰을 보관하지 않습니다. 각 CLI의 인증은 해당 도구의 표준 방식(설정 파일 / 환경변수)을 그대로 사용합니다.
